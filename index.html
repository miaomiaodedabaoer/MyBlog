Redis 的 String

保存整数

当保存 64 位有符号整数 时，String 类型会把它保存为一个 8 字节的 Long 类型整数，这种方式成为 int 编码方式；

包含字符

当保存的数据中 包含字符 时，String 类型使用 SDS （Simple dynamic string）简单动态结构体保存。

一个 SDS 由三部分组成，buf 保存实际数据（末尾加 ‘\0’, 和 C 一样 ）， len （4B）表示 buf 已用长度， alloc（4B）表示 buf 的实际分配长度。

除了 SDS 的开销，redis 需要用一个 RedisObject 来记录 SDS 的 元数据（最后一次访问时间，被引用次数等）； RedisObject 包含了 8B 的元数据和 一个 8B 的 指针 指向数据的实际位置。如果保存的是 Long 类型整数的话，为了提高效率，redisObject 的指针直接赋值为 整数数据。

Redis 使用一个全局哈希表保存所有键值对，哈希表的每一项是一个 dictEntry 结构体，用来指向一个键值对。dictEntry 中有三个 8B 的指针，分别指向 Key， value 和下一个 dictEntry。But，redis 使用的 内存分配库 jemalloc 在分配内存时，假设申请 N 个字节，实际会分配一个比 N 大的且为 2 的幂次方的空间，这样可以减少分配次数。

以上基本就是 redis 中 string 的存储方式，现在假设我们要存储一个键值对，key = 12345678, value = 123123; 如果以字符串方式存储，因为 key 和value 可以作为 Long 类型存储，那共需要两个 16B 的 redisObject 和一个 32B 的 dictEntry，共 64B，而我们实际存储的数据只是两个 8B 的 long类型整数，有 75% 的空间都存储一些乱七八糟的数据，效率比较底下。

压缩列表 ziplist

压缩列表是 redis 底层的一种数据结构，比较节省内存。ziplist 的表头有三个字段 ，表尾有一个字段，中间存储的是一个一个的 entry

zlbytes ： 表示列表长度； zltail 表示 表尾偏移量； zllen 表示列表中 entry 个数。表尾还有一个 zlend 字段表示列表结束。

entry 的内存结构分为 4 部分，

prev_len：上一个 entry 长度，取值 1B （上一个小于 254 字节）或 *5B *（大于254）；

len： 自身长度， 4B；

encoding： 表示编码方式 1B

content： 保存实际数据。

这样一来，如果存储一个 long 类型 value，只需要 1+4+1+8 = 14B，实际分配 16B （jemalloc）。

还是回到刚刚例子，key = 12345678, value = 123123; 假设我们用压缩列表的一个 entry 花费 16B 存储了这个 value，但是我们怎么根据 key 找到它呢。其实就是说怎么 用集合类型保存键值对。

只用把 key 拆分为两部分，

这也就是 Hash 类型的 二级编码 方法。就是把一个单值的数据拆分成两部分，前一部分作为 Hash 集合的 key，后一部分作为 Hash 集合的 field, value 就是 value。

而 Redis Hash 类型有两种底层实现结构，分别是压缩列表（节省空间）和哈希表。Hash 类型设置了用 ziplist 保存数据时的两个阈值，超过任意一个阈值，Hash 类型就会用哈希表来保存数据。

hash-max-ziplist-entries: 压缩列表保存时哈希集合中的 最大元素个数。

hash-max-ziplist-value: 压缩列表保存时哈希集合中 单个元素的最大长度。

这两个参数需要注意，因为如果转为哈希表，Hash 类型就会一直用哈希表进行保存，而不会再转回压缩列表了。

主从同步与哨兵

为了保证高可靠，redis 增加了一些从库与主库配合使用，确保在主库宕机时不过大影响服务。主从库之间采用的是读写分离的方式。主库、从库都可以接收 读操作; 而 写操作 要先到主库执行，然后主库将写操作同步给从库，从而确保主从库数据的一致性，如果都能写容易乱套，加锁的话会带来很大开销。

主从同步

首先，建立连接，从库向发送主库 psync 命令，参数包含 runID（主库id，不知道就是 ？） 和复制进度 offset（第一次为 -1，断连恢复时设置为其他值） ；主库收到后 回复 FULLRESYNC，全量复制命令，带上主库 runID 和 主库目前复制进度。

然后，主库执行 bgsave 命令，生成并发送 RDB 文件给从库（这个过程中主库仍接受 写操作，把这些操作记录到 replication buffer 里面）。从库收到 RDB 文件后，先清空其数据库，再加载 RDB，从而和主库保持一致。

最后，主库把 replication buffer 的内容发给从库，从库再执行这些操作。

在一次全量复制中，主库需要生成 RDB 文件（fork 子进程会阻塞主线程），传输 RDB（占用带宽），为了减轻主库的压力，可以采取 主从级联模式 主-从-从 的模式，选出一个从库作为小领导，把其他一些从库和小领导建立主从关系，这些从库在进行同步时直接和小领导联系，不去打扰主库。

主从库完成了全量复制，它们之间就会一直维护一个网络连接，主库会通过这个连接将后续收到的命令操作再同步给从库，这个过程也称为基于长连接的命令传播，可以避免频繁建立连接的开销。

断连后的恢复

Redis 2.8 开始，网络断了之后，主从库会采用增量复制的方式继续同步。

主库会把断连期间收到的写操作命令，写入 replication buffer，同时也写入 repl_backlog_buffer 这个缓冲区。repl_backlog_buffer 是一个环形缓冲区，主库会记录自己写到的位置，从库则会记录自己读到的位置。主从库对应的参数分别是 master_repl_offset, slave_repl_offset。连接恢复后，从库就会把其 offeset 发送到主库，主库与自己的offset 判断后就会把断连期间的操作传给从库。

real_backlog_buffer 是一个环形内存区域，如果断连期间写操作过多，就可能导致从库还没有读取到的数据被覆盖，因此我们需要综合考虑主库命令写入速率和命令大小，以及 网络传输速率 来调整该参数的大小。

哨兵

为了保证主从库的可用性（处理主库挂了），redis 引入了哨兵的机制，哨兵也是 redis 的实例，只不过不接受数据读写请求，而致力于 监控，选主，通知

监控

哨兵会周期性给主从库发送 PING 命令，来监控是否库是否正常，如果响应超时，谨慎起见（网络拥堵，主库压力大等其他原因导致超时），哨兵会把该库先标记为 “主观下线”，然后询问其他哨兵（哨兵一般为一个集群），只有认为该库下线的哨兵数量多于一个阈值时（N/2 + 1， N 为哨兵总数），才会把该库标记为“客观下线”

选主

判断主库下线后，开启选主阶段。

首先，选出仍然在线的从库，并检测它之前的网络状态（与主库断连的次数）得到第一次筛选结果

然后，选择 优先级（可以手动设置，如内存大小） 最高的作为主库，如果选出多个，再根据 复制进度（选与主库接近的），如果再有一样的，就选 ID 号小的。

通知

RDB 内存快照

AOF 的好处是，每次执行时只记录操作命令，需要持久化的数据量不大，对性能不会造成太大影响。问题是，在进行故障恢复时，redis 需要把日志上的命令逐一执行，如果操作日志很多，redis就会恢复的很慢，影响正常使用。

而 redis 的另一种恢复机制 内存快照，相当于记录某一时刻内存中数据的状态，这个快照文件就称为 RDB （Redis Database）文件，在数据恢复时，直接把 RDB 文件读入内存，实现 快速恢复；

但是这张快照的拍摄要注意：

拍多少

提供所有数据的可靠性保证，reids 执行的是全量快照，redis 提供两个命令来生成 （阻塞主线程与否）

​ save： 主线程中执行，阻塞

​ bgsave：创建子线程，专门用于写入 RDB 文件，默认配置，非阻塞

人动怎么办

制作快照过程中，主线程仍然可以处理写操作，redis 借助操作系统提供的 写时复制（主线程在有写操作时，才会把这个新写或修改后的数据 写入到一个新的物理地址中，并修改自己的页表映射） 技术生成该数据的副本。然后，bgsave 子进程会把这个副本数据写入 RDB 文件，而在这个过程中，主线程仍然可以直接修改原来的数据。

拍摄频率

与 AOF 类似，拍的越频繁，宕机时丢失数据内容越少，但是也越影响性能，因为 fork 子线程时会阻塞主线程，并且也会给磁盘带来压力。那么我们在第一次拍摄全景之后，每次拍摄只拍摄动的东西（也就是被修改的数据）不就好了，为了记住那些数据做了修改，我们可以用 AOF 日志进行记录（Redis 4.0 提出，混合策略）

AOF 和 RDB 的选择取决于 丢失容忍度 和 性能 两方面的考虑。如果数据不能丢失，可以结合内存快照和 AOF; 如果允许分钟级别的数据丢失，可以只使用 RDB; 如果只用 AOF，优先使用 everysec 的配置选项，因为它在可靠性和性能之间取了一个 平衡。

Redis AOF 日志

为了避免宕机后 Redis 数据丢失， redis 使用了 AOF 和 RDB 两种机制。

AOF 日志 （append only file）

写后日志

不像其他数据库的写前日志，Redis 是先执行命令，把数据写入 内存，然后才把 操作命令 记录日志（以文件的形式）。这样一来可以避免记录错误日志，并且操作完成后再记录可以 避免阻塞线程。

风险

AOF 也存在两个风险。首先，如果刚执行完一个命令，还没有来得及记日志就宕机了，那么这个命令和相应的数据就有 丢失 的风险。

其次，AOF 可能会给下一个操作带来阻塞。这是因为，AOF 日志也是在主线程中执行的，如果在把日志文件写入磁盘时很慢，就会阻塞线程。

因此，如果我们能够控制一个写命令执行完后 AOF 日志写回磁盘的 时机，这两个风险就解除了。

三种写回策略

Always 同步写回

每个写命令执行完，立马同步地将日志写回磁盘；基本可以不丢数据，但是给每次操作带来了额外的负担，会影响主线程性能

Everysec 每秒写回

每个写命令执行完，先把日志写到 AOF 文件的内存缓冲区，每隔一秒把缓冲区中的内容写入磁盘；避免了同步写会那么大的开销，丢失数据最多丢 1 秒。

No 操作系统控制的写回

写命令执行完后，先把日志写到 AOF 文件的内存缓冲区，由操作系统决定何时将缓冲区内容写回磁盘。释放控制权，如果 AOF 没有写回磁盘，宕机后数据丢失可能会很多。

这三种策略在 可靠性 和 性能 上依次下降和增加

AOF 重写

AOF 是以文件的形式记录接收到的所有 写命令。随着接收的写命令越来越多，AOF 文件会越来越大，会带来一些问题：

  1. 文件系统大小限制
  2. 大文件追加记录效率低
  3. 宕机后恢复过程缓慢

AOF 重写机制就是在重写时，Redis 根据数据库的现状 创建一个新的 AOF 文件（ 读取数据库中的所有键值对，然后对每一个键值对用一条命令记录它的写入），这样可以实现 多变一 的效果。

重写过程

和 AOF 日志由主线程写回不同，重写过程是由后台线程 bgrewriteaof 来完成的，这也是为了避免阻塞主线程，导致数据库性能下降。

首先，每次执行重写时，主线程 fork 出后台的 bgrewriteaof 子进程。此时，fork 会把主线程的 内存拷贝 一份（通过页表）给 bgrewriteaof 子进程，这里面就包含了数据库的最新数据。

这个过程中，同时存在 两份日志：第一处日志就是指 正在使用的 AOF 日志，Redis 会把后台线程复制时进来的写操作写到它的缓冲区。这样可以保证在这段时间内的宕机可以正常恢复，

第二处日志就是新的 AOF 重写日志。这些新来的操作也会被写到重写日志的缓冲区。等到拷贝数据的所有操作记录重写完成后，这些最新操作也会写入新的 AOF 文件，这时就可以用新的 AOF 文件替代旧文件了。

Api styles

API 规定了网络主体中通信的一些规则，不同的 API 风格适用于不同的业务和情景，今天在搜索论文相关东西时候，看到了主要的 5 种 API 风格

  1. Tunnel Style:

    这种风格的 API 是一些 fuction 的集合，这些方法可以被远程调用；比较出名的就是 gRPC。

  2. Resource Style

    API 是资源的集合，可以和消费者进行特定种类的交互；提供者对外界暴露一些资源，最常见的 就是 OpenAPI （将服务封装成一些列 API 开放出去供人调用）

  3. Hypermedia Style:

    和上面类似，这类 API 是一些 相互关联的资源的集合，就像 Web 上的资源一样；基于资源，基于link，我们请求这些资源，并不是一个独立的调用，得到的资源又包含别的资源的路径，这样就可以组成一个工作流。

  4. Query Style

    这类 API 公开了一个结构化数据模型，通过通用查询语言进行查询和更新。如 facebook 的 graphQL, 与消费者的通信大多是对数据库的查询更新，如 CRUD users, posts, likes ,mentions, query 风格的API 可以有效的请求这些数据。

  5. Event-based Style

    API 是一些事件的集合，这些事件由提供者发布并且可以被消费者订阅。它逆转了。大部分使用 pub/sub模型，API 的使用者会监听事件，收到他们所订阅的特定事件的通知，提供者只需要将事件发布到特定地方，无需关注消费主体。

Wireshark下HTTPS通信过程

按照 《图解HTTP》上的定义， HTTPS 的通信步骤如下，1-4 步

步骤 1： 客户端通过发送 Client Hello 报文开始 SSL 通信。报文中包 含客户端支持的 SSL 的指定版本、加密组件（Cipher Suite）列表（所 使用的加密算法及密钥长度等）。

步骤 2： 服务器可进行 SSL 通信时，会以 Server Hello 报文作为应答。和客户端一样，在报文中包含 SSL 版本以及加密组件。服务器的 加密组件内容是从接收到的客户端加密组件内筛选出来的。

步骤 3： 之后服务器发送 Certificate 报文。报文中包含公开密钥证 书。

步骤 4： 最后服务器发送 Server Hello Done 报文通知客户端，最初阶段的 SSL 握手协商部分结束。

步骤 5： SSL 第一次握手结束之后，客户端以 Client Key Exchange 报 文作为回应。报文中包含通信加密中使用的一种被称为 Pre-master secret 的随机密码串。该报文已用步骤 3 中的公开密钥进行加密。

步骤 6： 接着客户端继续发送 Change Cipher Spec 报文。该报文会提 示服务器，在此报文之后的通信会采用 Pre-master secret 密钥加密。

步骤 7： 客户端发送 Finished 报文。该报文包含连接至今全部报文的 整体校验值。这次握手协商是否能够成功，要以服务器是否能够正确 解密该报文作为判定标准。

步骤 8： 服务器同样发送 Change Cipher Spec 报文。

步骤 9： 服务器同样发送 Finished 报文。

步骤 10： 服务器和客户端的 Finished 报文交换完毕之后，SSL 连接 就算建立完成。当然，通信会受到 SSL 的保护。从此处开始进行应用 层协议的通信，即发送 HTTP 请求。

步骤 11： 应用层协议通信，即发送 HTTP 响应。

步骤 12： 最后由客户端断开连接。断开连接时，发送 close_notify 报 文。下图做了一些省略，这步之后再发送 TCP FIN 报文来关闭与 TCP 的通信。

迭代遍历二叉树

二叉树的遍历从深度遍历来看，分为前序遍历，中序遍历和后序遍历，其中的前中后指 根节点 在遍历时的相对位置，前序就是指 根 - 左 - 右，中序指 左 - 根 - 右；

使用递归遍历二叉树十分简单, 中序遍历代码如下，前序和后序只需改变最后三行代码顺序即可。

使用迭代遍历的话，首先想到借助 栈 模拟递归，其次在纸上设计一下节点应该怎么 入栈，取值，出栈， 即可；

中序遍历

以中序遍历为例子，左 - 根 - 右 的方式下:

  1. 把所有 左节点 全部 入栈，直到没有 右边的非NULL节点

  2. 取出最顶端节点（该节点就是树形图中最左下的元素，也就是应该第一个取到值的），加入列表

  3. 此时 左 - 根 - 右 中的 左，根 节点已经遍历完成，对右节点重复上两步操作即可。

仔细看来，中序遍历的迭代遍历代码和递归有神似之处，划分为3块。

前序遍历

前序遍历的顺序为 根 - 左- 右，迭代遍历可以分为三步

  1. 取出栈顶元素（视为根元素），进行记录

  2. 把不是 NULL 的 子元素入栈，但是 右先左后 （这样出栈时才是 根 - 左 - 右）

  3. 重复上两步，直到栈空

后序遍历

把前序遍历的代码做两处修改即后序遍历，首先是

  1. 列表中 从后往前 添加
  2. 子元素 反序入栈

Https

HTTP 的问题

从 HTTP 协议本身来看，其存在几大问题：

可能被窃听

HTTP 本身不具备加密的功能，在互联网的线路上传输时，可能会被人收集，这样就会看到通信内容。

解决： 对信道加密（结合 SSL 和 TLS）； 对内容加密

内容可能被篡改

HTTP 请求或响应在传输途中，可能遭到攻击者拦截并篡改，这种方式称为中间人攻击（Man-In-Middle attack).

不验证身份

只要知道 IP 和端口，任何人都可以发起请求，服务器也会响应。解决：使用 SSL ，查看证书 CA， 证书由第三方颁发，客户端通信前先确认证书。

HTTPS

https 的出现正是解决了这些问题，https 并不是一种新的协议，只是 HTTP 通信接口部分用 SSL（Secure Socket Layer）和 TLS
( Transport Layer Security)，其和 http 的区别简单来说就是：

HTTP + 加密 + 完整性保护 + 身份认证 = HTTPS

SSL 的加密

SSL 的加密结合了 共享密钥 和 公开密钥。公开密钥更安全，但是通信效率比较低；因此 SSL 先使用 公开密钥加密方式来传输在后面的共享密钥加密使用的密钥，确保交换的密钥安全后，使用共享密钥加密方式进行通信。

但是，在第一次通信时，客户端无法证明公开密钥本身的真实性；为解决这种问题，服务器可以向数字证书认证机构申请一个数字签名，在通信时一并发给客户端，客户端拿着签名去验证即可。

HTTPS 缺点

  1. 耗费资源

    与纯文本通信相比，加密通信会消耗服务器更多的资源

  2. 速度慢

    多了 SSL 通信部分，还有加密解密部分，整体速度比HTTP慢 2 到 100倍

  3. 证书贵

    要进行 HTTPS 通信，证书是必不可少的。而使用的证书必须向认 证机构（CA）购买。因此一些个人网站，可能无法负担这一开销

各种加密方式

共享密钥加密

接收方和发送方使用同一把密钥进行解密和加密工作，发送方要把密钥发给接收方，在这个过程中 密钥 可能会泄漏，但是这种方式 加密，解密速度快

公开密钥加密

接收方生成，公钥和私钥，公钥发给发送方，发送方使用 公钥 进行 加密，再把密文发送给接收方，接收方使用 私钥 进行 解密。

保证了 密钥 传递的安全，但是解密加密较慢，并且 公开密钥 在传递过程中可能被中间人截取，给发送方发一个错误的，中间人使用正确的公钥 加密 错误的信息给接收方，发送方，接收方并不知情（中间人攻击）

混合密钥加密

使用公开密钥的方式 传递 共享密钥，确保 共享密钥 传输正确，之后用共享密钥来通信。

消息认证码

防止密文被篡改，使用一个共享密钥对信息进行计算，得到（类似哈希值） MAC（message authentication code) 连同这个码发给接收方，接收方根据同样的密钥进行解密

但是，由于两方都可以自己生成 MAC，这个信息是自己伪造的还是真正发过来的没办法识别，

数字签名

数字签名可以解决该事后否认的问题：发送者在发送时给消息加上数字签名，数字签名的生成使用公开密钥加密，发送者通过私钥生成，接收者使用公钥对签名进行解密，这样就可以确认消息的发送者。

但是无法保证公开密钥确实来自发送者，有可能被中间人替换

数字证书

认证中心通过自己的私钥，对发送者的资料（发送者的公钥已经个人信息）生成数字签名，加上资料放到数字证书里，发给发送者，发送者再发给接收者，接收者通过认证中心的公钥对证书内对签名进行认证，来确认确实来自发送者。

JWT

是什么

JSON Web Token （JWT）是一个 开放的 标准，以紧凑且 self-contained 的方式，使用 JSON 对象在 parties 间安全的传输信息。这些信息是可以被验证且信任的，JWT 可以使用一个 secret （借助HMAC算法）或者一个 public/private 钥匙 （借助 RSA 或者 ECDSA）

JWT 主要用于 签名 token，签名 token 可以供他人验证其中包含的信息（claims）的完整性

什么时候用

1. 用户授权

因为 HTTP 是开放的，且无状态，server 不知道用户的身份，就要做访问权限的控制，认证用户后，才允许调用 API。具体是，用户第一次登陆时，返回一个 JWT，里面包含用户的一些信息，用户之后的所有请求都带着 JWT，验证后允许其访问一些路径，服务，资源。使用 Token 的好处是不用每次都发送 账号密码，减小风险。

由于 JWT 自身较小的开销？和 轻松跨域，Single Sing On 广泛应用了 JWT。

2. 信息交换

由于 JWT 可以被签名，使用 public key/ private key可以确定发送人的身份；并且，使用 JWT 内容中的 header 和 payload, 可以轻松的验证内容是否被篡改。

结构

两种表现形式 JSON 和 压缩后的，都由三部分组成，

三部分： 名字都是 3个 character (宣传为了紧凑)

  • Header, （ token 的类型，签名算法）

    { "alg": "HS256", "typ": "JWT" }

  • Payload （携带的有用的一些信息 claims，3类,

    1. Registered claims: 推荐的，预定义的属性，包括 iss (issuer), (expiration time), sub (subject), aud (audience)

    2. Public claims: 由使用 JWT 的人随意使用，尽量与标准中定义的不冲突

    3. Private claims: 客制化定义，分享信息的人定义，更加私人化

      由于这些信息都是可读的，不建议在里面放一些 机密信息，除非他们是加密的，

      { "sub": "1234567890", "name": "John Doe", "admin": true }

  • Signature （ 由以上两部分，以及你的 secret，使用 header中声明的算法，进行签名）

    HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)

把 header 和 payload 分别用 Base64url 编码，再加上 signature，三部分以点分隔就形成了压缩后的结构：xxxxx.yyyyy.zzzzz 。

使用

无状态的验证机制，server 并不存储 JWT，当用户成功登陆时，返回一个 JWT，用户之后的请求都带上，使用 Bearer schema 放到 Authorization header 中，这样 CORS （Cross-Origin Resource Sharing）跨域问题就得到了解决。

「 JWT 数据可以放到 HTTP 请求头部发送，不存在跨域问题。而原生 session 基于 cookie 发送，存在跨域问题」

为啥 JWT 呢

官网上与 Simple Web Tokens (SWT) ，Security Assertion Markup Language Tokens (SAML) 做了比较。

物理属性方面，对比与 SAML，JSON没有 XML 冗长，并且使用 JSON 编码起来体积更小，利于在http中传输

安全性方面，对比 SWT，SWT 只能使用 HMAC 算法进行对称加密，JWT 和 SMAL 可以用 公钥/私钥进行签名，SMAL在这方面又比较困难。

与 Session的比较

主要区别就是存储的位置， session 保存在服务端，jwt 在客户端

  1. 安全性

    jwt 使用 base64 编码，并没有加密，不能存放机密数据，而session在服务端，可以放敏感数据，更安全

  2. 性能

    jwt 太长，cookie 一般限制 4k，一般放在 localstorge 里面，放置在 HTTP 的请求头中，就导致 Header 可能比 Body 还大

    「超过了tomcat允许的最大值。默认情况下，tomcat（8.0版本）允许的http请求header的最大值是8024个字节（8KB）」

  3. 一次性

    无法废弃，续签

因此，最好设置的有效时间短一点，只希望被使用 1次。网上看到的例子，激活账户的邮件，几小时内有效，标识该用户，无法篡改。

内存管理

内存管理

内存是计算机中需要认真管理的重要资源，尤其是分层存储器体系中空间较小且昂贵的资源，如高速缓存，RAM，操作系统中管理分层存储器体系的部分称为存储管理器。他的主要任务是记录空闲的和正在使用的内存，为用户分配内存，用户使用完后及时释放内存。

无存储器抽象

最简单的存储器抽象就是没有抽象，每一个程序都直接访问物理内存。如果用户可以寻址内存的每个字节，就很可能破坏操作系统，并且多个程序运行过程中也会互相影响。

一种存储器抽象：地址空间

地址空间是一个进程可用于寻址内存的一套地址集合，每一个进程都有一个自己的地址空间，并且不同进程的地址空间相互独立。

那么如何保证两个进程在访问它们地址空间中的相同位置（如进程1的地址28和进程2的地址28）不会冲突呢？

我们需要使用动态重定位，简单地把每个进程的地址空间映射到物理内存的不同部分，可以简单的给每个CPU配置 基址寄存器 和 界限寄存器。当一个程序运行时，将他的起始物理地址装载到基址寄存器中，程序的长度装载到界限寄存器中，在进行寻址时，将地址空间的地址加上基址寄存器的值，就可以得到真实的物理地址，如果大于程序的最大长度，就会产生错误并中止访问。

这种方式的缺点是每次访问内存都要进行加法和比较运算。加法运算由于进位传递时间的问题，会比较耗费时间。

交换技术

当计算机的内存不够使用时，我们可以使用交换技术，把运行一段时间的进程存到磁盘里，让他们进入睡眠状态，然后从磁盘中唤醒一些进程放入内存。

因为进程在运行过程中所占用的内存空间有可能增长，在换入进程时要为他分配一些额外的内存，以免他增长的空间影响到其他进程；而将进程移动到磁盘里进入睡眠时，应仅仅分配内存实际使用的内存，将额外的内存交换出去是一种浪费。

空闲内存分配

我们可以使用 位图 进行存储管理，位图将存储空间分为一个个存储单元，对存储单元进行编码，0代表空虚，1代表占用（也可相反），如下图所示，ABCDE为5个进程，刻度代表内存单位，阴影部分表示空闲区域，编码为0。也可以采用链表的方式表示同样的信息。

位图的大小取决于内存空间的大小以及内存单元的粒度（细腻程度，也就是大小），这种方法主要问题是，在寻找空闲空间时要在位图中搜索连续 k 个 0 的串，这将十分耗时，并且该串还可能跨越字的边界。

另一种记录内存的方法是，维护记录已分配内存和空闲 内存段链表。链表中的节点为进程或者空闲区。段链表可以按照地址排序，这样更新起来非常直接，如图所示，在进程x终止后，内存段链表的更新就只是相邻节点的合并或连接。

当按照地址顺序分配内存时，有几种算法，第一个是首次适配算法，沿着链表遍历，直到第一次找到合适的空闲位置，就分配。将每次找到的空闲位置都记录下来，下次就接着结束的位置找，叫做下次适配算法；

另一个著名并广泛应用的是最佳适配算法，搜索整个链表，找出能容纳进程的最小空间，他趋向于找到一个最接近实际需要的空闲区，而不是随便找到一个就用，不会把潜在的较大空闲区拆分。但是这种方法较慢，并且比以上两种更浪费内存，因为他会产生很多非常小的空闲区。为避免这种问题，可以采用最差适配，即总是分配最大的可用空闲区，使新的空闲区比较大而可以继续使用。

还有一种分配算法称为快速适配，它为那些常用大小的空闲区维护单独的链表。这样可以快速的找到制定大小的空闲区，但是问题在于当进程被换出时，寻找它相邻块并查看是否可以合并的过程是非常费时的。

虚拟内存

当一个程序占用的内存大于计算机内存时，我们需要把该程序分割成许多个片段，每次只加载重要的一些片段，系统对内存中的片段和硬盘中的片段时不时的进行换入换出操作。这种方法叫做虚拟内存。这些独立的块或者片段称为页或 页面。

分页

大部分虚拟内存系统中都使用一种叫做分页的技术。在程序中产生的地址称为虚拟地址，它们构成了一个虚拟地址空间。在使用虚拟内存时，CPU获得一个地址后，不直接把地址送到内存总线上（因为并不是真正的物理地址），而被送到内存管理单元（MMU），由它来映射至物理内存地址。物理内存中的 *页框 *对应着虚拟内存中的页面。RAM和磁盘直接的交换总是以整个页面为单位进行的。

MMU 的内部维护着一个页表，其中包含了虚拟地址和物理地址的映射关系，如下图所示，虚拟页面1-7分别对应着几个物理内存（这些程序片段已经被加载进内存，因此有物理地址）。需要注意的是，因为不是所以页面都被加载进内存，这里并不是所有虚拟页面都有物理内存与其对应，例如 所有的虚拟内存片段 x。

那么如果现在我们想寻找某个虚拟页面X的物理地址，MMU会注意到这个地址没有被映射，于是CPU陷入到操作系统，称为缺页中断，OS选择一个很少被使用的页框，将他的内容写进磁盘，然后把需要访问的页面放到刚刚回收的页框里，修改映射关系，然后重新启动引起陷阱的指令。

下面是MMU的具体工作原理，下图的最下方是输入的虚拟内存指令，高4位是代表页号，低12位代表偏移量。而页表中每个表项的高三位代表对应的物理地址中最高的三位，最低的一位代表该地址是否在内存中。首先根据寻找对应页号的页表项，如下图所示找到 2 号表项，将虚拟地址的最高三位替换，保留剩下的位，即可得到物理地址。

页表项中还可以加入其他含有特定信息的位，如

  • 保护位：指出该页是否允许读，写和执行。
  • 修改位：当一个表项被修改过时，将该位标记，当表框被换出时，OS可以决定是简单的丢弃还是重新写入磁盘。
  • 访问位：在发生缺页中断时帮助OS选择该淘汰的页面，不再使用的页面更适合淘汰。
  • 高速缓存禁止位：在一些情况下保证硬件不断的从设备中读取数据，而不是访问高速缓存中旧的数据。

加速分页过程

在实现虚拟内存以及分页时，我们需要考虑映射速度，以及页表大小的问题。现代计算机至少有32位的虚拟地址，假设页面大小位4KB，32位的地址空间将有100万页，如何在这么大的页表里进行查询是个问题。

一种方法是我们把每个进程的页表放到寄存器中，这样查询起来很快，但是在每次上下文切换也就是进程切换时会重新装载整个页表，代价高昂。另一种方法，那就放内存里白，但是这样速度又太慢。

因此，还是那个老办法，缓存吧，给计算机加一个小硬件，转化检测缓冲区（Translation Lookaside Buffer），也叫快表，通常在MMU中，他存储少量但是经常被查询的表项，工作原理和高速缓存类似。也可以使用软件来模拟TLB，通过 TLB 找不到物理内存时称为TLB失效。

  • 软失效：该页面不在TLB中，在内存中。这时更新一下TLB即可，耗时10—20个机器指令，几纳秒。
  • 硬失效：不在TLB也不在内存中，这时需要通过磁盘中存取页面，并且需要页表遍历，耗时往往时软失效的百万倍。
  • 次要缺页错误：所需页面就在内存中，但是没有记录在该进程的页表中，如该页面可能由其他进程从硬盘中调用内存，这时需要把所需页面正确映射到页表，不在从硬盘中调入。
  • 严重缺页错误：需要从硬盘重新调入页面。
  • 段错误：程序访问了一个非法地址，OS一般会报告段错误来终止程序。

针对大内存的页表

针对非常大的内存，我们可以使用多极页表的形式，高级页表的表项指向低级页表的地址，类似套娃的形式，这样可以避免把全部页表一直保持在内存中。

不使用套娃的方法是倒排页表，这种设计中。实际内存的每个页框对应一个表项（而不是虚拟内存页对应表项），由于虚拟内存一般都比实际内存大很多，这样的方式可以节省大量的空间。

但是问题是这样查询起来比较麻烦，当进程 n访问虚拟页面 p 时，不能仅仅通过 p 来找物理地址了，而需要遍历页表来查询表项 (n,p)。解决方法还是 TLB，把频繁使用的页面存起来，查的就快一点。还可以对整个页表建立散列表，当TLB失效时，查询起来就很快。

页面置换算法

当发生缺页中断时，我们需要选择一个页面将其换出内存，以便腾出空间分配给新页面。当我们需要考虑到底选择换出哪些页面，需要考虑到页面在一段时间内还会被访问的可能性，如果换出被频繁访问的页面，就会降低系统性能，接下来介绍一些算法。

最优页面置换

最理想的情况是我们知道内存中各个页面下一次访问的时间，有些页面可以几条指令后就会被访问，有些则要经过几百万次，因此我们每次都换出下一次访问时间最长的。但是这种方法是不可能实现的，我们无法预知页面的下一次访问时间。

最近未使用页面置换 NRU

在页面被读取或者写入时设置 R 位，如果页面被修改，再设置 M 位。因此系统可以得知哪些页面在短时间内可能再被访问，避免置换出去。当发生缺页中断时，操作系统检测所有页面，根据它们 R 位和 M 位的值，把它们分为 4 类：

  1. 没有被访问，没有被修改
  2. 没有被访问，已被修改
  3. 被访问，没有被修改
  4. 被访问，被修改

NRU（Not Recently Used）算法随机从类编号最小的页面中挑选一个页面淘汰，这种方法易于理解并且能够有效的实现。

先进先出页面置换算法

另一种开销较小的页面置换算法是 FIFO，原理是找到存在时间最长的页面并淘汰他，操作系统维护一个页面链表（或者是队列），将新进入的放在表尾，老的就被挤到表头，每次淘汰表头的页面，但是这样做可能会把频繁使用的页面置换出去。

第二次机会页面置换算法

这种算法对FIFO进行优化，检测最老的页面的R位，如果R位是0，代表又老又没人访问，就立刻置换出去，如果是1就把R位清零，并把该页面放到链表的尾端，也就是给他第二次机会。

时钟页面置换算法

第二次机会算法虽然不错，但是每次移动页面会降低效率，完全可以用循环链表完成，使用一个表针指向最老的页面，当发生缺页中断时，检测表针指向的页面，如果是 0 ，就淘汰，如果是 1，就清零并向前移动一格，重复这个过程知道找到一个 R 为0的页面为止。

最近最少使用页面置换算法

LRU（Least Recently Used）致力于实现最优置换方法，通过记录各个页面在前几条指令中频繁使用的页面，这些页面很可能在之后的指令还要访问，这样每次置换时就淘汰那些未访问时间最长的页面。

但是这种方法实现代价很高，在内存中维护一个所有页面的链表，最近使用最多的在表头，最少的在表位，而每次访问内存时都要更新整个链表。

一种解决方式是在硬件中加入一个64位的计数器，每次执行一条指令都加1，页面项中加入一个域都记录这个值，发生缺页中断时，选择值最小，也就是最老的那个页面淘汰。

NFU

大部分电脑都没有这种硬件，一直软件解决方案 NFU（Not Frequently Used）给每个页面分配一个计数器，在每次缺页中断时，如果页面的 R 位为1，就给他的计数器加1。这样系统就大致掌握了每个页面被访问的频繁程度，发生中断时，置换计数器值最小的页面。

对其的一种优化方法称为老化算法，在 R位加进计数器之前先将计数器右移一位，再将 R 位加到计数器最左边，发生缺页中断时还是置换值最小的页面。相比于之前的算法，这样的好处是，在两个页面访问次数相同时，可以判断哪个访问时间较早，

工作集页面置换算法

一个进程正在使用的页面的集合称为它的工作集，不少分页系统都会跟踪进程的工作集，以确保让进程运行前，它的工作集就已在内存中，这样可以大大减少缺页中断的概率。

将每一个页面项设置上次访问时间和 R 位，发生缺页中断时，先扫描 R位如果等于 1，更新页面访问时间位当前时间，如果R位0，并且页面项纪录的访问时间和当前时间之差大于预设的一个数，就淘汰，如果小于等于，就把这一项作为候选项，整个扫描完成后如果没有淘汰页面，就在这些候选项中选一个未访问时间最长的。

工作集时钟页面置换算法

还是老问题，每次都扫描一个效率太低了，还是基于时钟算法。每次缺页中断时，首先检查R位，如果是1，该页面刚刚就被访问过，不适合淘汰，把R置0，指针移动一个，如果是0，如果页面的生存时间大于一个值，并且他没有被访问过，并在磁盘上有一个有效的副本，就把他置换出去，而如果被修改过，则不适合被置换，因为还要修改其磁盘上的副本，这时指针继续向前走，重复该过程。

总之，最好的两种算法是老化算法和工作集时钟算法，它们分别基于 LRU 和 工作集，一种纪录页面被访问的总次数，另一种直接纪录短时间内是否被访问，都具有良好的页面调度性能也可以有效的实现。

分页系统中的设计问题

上面的内容仅仅讲述了一些基本的页面置换算法和如何实现它们，但是分页系统还要考虑一些事情。

局部分配与全局分配

内存中运行着不止一个进程，因此我们在进行页面置换时，完全可以分配那些其他进程中可以淘汰掉的页面，分配给每个进程的页框数是随时间 动态 变化的，这类算法称为 全局 页面置换算法。

全局算法通常比局部算法更好，在工作集大小动态变化的过程中，即使有大量空闲页框存在，工作集的增长也会导致颠簸，如果分配的页框较多，也会造成空间浪费。

在我们之前介绍过的算法中，大部分都适合于这两种方法，而工作集和WSClock算法只有采用局部策略才有意义。

页面大小

页面的大小需要平衡，如果分配的页面太大，那么进程的正文段，数据段或者堆栈段可能填不满一整个页面；并且在 TLB空间中，使用大页面可以占用更少的TLB表项，节省空间，小页面则更加精确。操作系统有时会位系统中不同部分使用不同的页面大小，内核使用大页面，用户进程使用小页面。

如果分配的页面太小，则会生成更大的页表，并且内存于磁盘会产生更多的交互，而传输的大部分时间会花费在寻道和旋转延迟上，所以传输大页面和小页面花费时间基本相同。

分段

目前为止讨论的虚拟内存都是一维的，虚拟地址连续的从0到最大，而对于内存动态变化的多进程来说，这可能会导致表与表之间发生碰撞。并且多个进程之间会产生一些闲置空间，造成浪费。因此引出了分段的思想。

在机器上提供多个互相独立的称为段的地址空间，每个段从0到最大的线性地址序列构成，段的大小在运行期间可以动态改变。

这样做的好处有3点，首先是段和段之间是独立增长和减小的，不会影响到其他的段，并且如果一个段被修改并重新编译，其他使用该段的段并不会受到影响，如果它们被放在一个连续的空间，则程序整体都要重新编译。并且分段有助于在不同进程间共享过程和数据。这样也有助于在几个进程间共享数据，一个常见的例子是共享库，可以把一些庞大的库放到一个单独的进程中由各个进程共享，从而不需要在各个进程的地址空间中都保存一份。其次，不同的段可以应用不同的保护模式，例如读写执行等权限。
